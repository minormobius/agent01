① What SpliceCraft is

SpliceCraft (by Binomica Labs, MIT-licensed) is "your whole cloning workflow, in the terminal" — a molecular-biology workbench built as a Textual TUI in 100% Python on top of BioPython. It does plasmid maps, restriction analysis, primer design, assembly planning (Golden Gate / MoClo / Gibson), local BLAST + HMM search, Sanger verification, in-silico PCR, gel simulation, and a lab notebook.

Crucially for a web port, it already ships a headless mode: a ~130-endpoint localhost JSON HTTP API (splicecraft --agent, bearer-token, 127.0.0.1-only, "single-tenant by design") plus a stdlib-only CLI sidecar. So there is a clean machine surface to build a frontend against — it just isn't multi-tenant or networked today.

Sources: repo · agent-api.md · pyproject.toml

② The core tension

minomobi ships on Cloudflare Pages + Workers — JS/WASM, no persistent Python compute. SpliceCraft is a stateful, single-tenant, native-dependency Python desktop app. There is no drop-in. Worse, the most valuable features lean on compiled C libraries that Pyodide cannot load (Pyodide has no wheels for pyhmmer or primer3-py), so "Python in the browser" loses exactly the hard parts.

The insight that unlocks everything: two of the three hard dependencies are already C. A Rust port can reuse them — bind them natively, or compile them to WASM via Emscripten — instead of reimplementing the science. That makes Rust→WASM a strictly better web target than Python→Pyodide.

③ Easy 90% vs hard 10%

Python the language is the easy thing to replace. From reading the source, most of SpliceCraft is ordinary application code with nothing language-specific:

  • Ports cleanly to Rust: the four-layer data-safety system (atomic writes, backups, shrink guard), JSON schema migration, the assembly planners, in-silico PCR, the gel migration model, the lab notebook, the --agent HTTP/JSON server, the string-level biology in splicecraft_biology.py (IUPAC regex, reverse-complement, circular slicing, wrap-aware feature lengths).
  • The TUI → ratatui. Labor, not difficulty.

The hard 10% is three scientific dependencies — but each is hard for a different reason, which changes the strategy:

⑤ Using the C libs from Rust — the mechanics

Rust isn't allergic to C. Reusing libprimer3 and libhmmer "as-is" is a solved problem; the only question is the build target. Both libraries expose a real C API (SpliceCraft links them via primer3-py/pyhmmer rather than shelling out to primer3_core/hmmscan binaries), so there's an API to bind, not a CLI to fork.

// build.rs — compile the vendored C and link it
fn main() {
    cc::Build::new()                // the `cc` crate drives the C compiler
        .files(glob_primer3_sources())
        .include("vendor/primer3/src")
        .compile("primer3");        // -> libprimer3.a, linked into the Rust binary
}

// bindgen turns primer3.h into `extern "C"` Rust declarations,
// which we wrap in a safe module. Then:
let tm = unsafe { ffi::oligotm(seq.as_ptr(), /* salt, conc, … */) };

Toolchain pieces: bindgen (header → bindings), the cc crate (compile the C in build.rs), and the Rustonomicon FFI chapter for the unsafe boundary. For HMMER you'd instead drive its autotools/CMake build from build.rs and link libhmmer + Easel.

⑥ Where it runs changes everything

"Use the C as-is" is free for one target and progressively harder for the others:

Two browser-WASM caveats worth flagging up front: COOP/COEP is the same cross-origin-isolation requirement labglass already lives with on this infra (labglass/_headers), so it's a known quantity on Pages; and nothing can shell out to a subprocess in WASM — fine here, because the lib-linking path is what we'd use.

⑦ The game layer

This may be the most important question: can we put a game over the useful tool? Yes — and it fits better than most "gamified tool" ideas, for one structural reason.

The tool is its own scoring oracle

Most science/edutainment games have to hand-author both the levels and the answer-checking. SpliceCraft hands you an objective correctness scorer for free: BioPython plus those 4,000 accuracy tests can deterministically judge any player move — is this a valid Golden Gate assembly? does this primer pair amplify the target? is the frame intact? are there internal cutters? The thing that normally kills these projects is solved before we start, which means levels can be procedurally generated and solutions auto-graded. That's what makes a game layer both cheap and deep. Precedent: Foldit, EteRNA, and Phylo all turned a scored biological problem into play.

Cloning is already a constraint-satisfaction puzzle

The wet-lab loop maps onto game mechanics almost 1:1. Constraints (restriction-site uniqueness, primer Tm windows, MoClo/Golden Gate overhang compatibility, reading frame, no internal cutters) are the puzzle rules; a valid plasmid is the win. In-silico PCR + gel sim give instant verifiable feedback — design → simulate → see the band → iterate. And optimization (fewest enzymes, fewest steps, lowest cost) gives Zachtronics-style replay value with leaderboard histograms.

The strategic kicker

Both modes we're putting on the roadmap run on the cheap Phase-2 deterministic engine (restriction, assembly, in-silico PCR, gel) — no primer3, no HMMER. So the game layer ships before the expensive C-lib port. The hard C dependencies (HMMER especially) are for the pro-tool, not the game. Genre choice is an engineering-cost choice, and we've picked the two cheapest-to-power, highest-synergy ones.

On the roadmap: two modes

ModeLoopEngineReuse

Zachtronics-style		 Given a target construct + a parts library, assemble a valid plasmid under real constraints. Scored on validity, then optimized for cost / steps / elegance with leaderboard histograms. Phase 2 Fluoddity's score / leaderboard / atlas shell

The directed-evolution mode is the tightest tie to existing work: it's Fluoddity's exact breed a genome → select by fitness → fork → publish as an ATProto record → phylogeny/atlas loop, but over real DNA with an objective fitness function instead of a Fourier black box. We reuse the whole shell — breeding UI, fork/phylogeny, ATProto persistence, the cross-user atlas (fluoddity.mino.mobi) — and swap the genome. (The "guess-the-gene" redact mode, à la rite's /redact, stays parked: it needs the expensive Phase-3 BLAST/HMM engine.)

Audience: tiered

A casual puzzle front-end that graduates into the real workbench. New players get an abstracted assembly puzzle that hides bookkeeping; as they level up, more of the real tool's surface (and real-construct export — GenBank you could actually take to a bench) is revealed. The game is the onboarding ramp to the tool, not a walled-off toy.

The one real design risk

Real cloning carries a lot of bookkeeping. The game layer's entire job is to hide the spreadsheet and surface the decisions — that's Zachtronics' actual craft. If a level exposes a parts-inventory table, it's a tool; if it exposes one interesting choice with consequences, it's a game. We make that a hard design constraint from level one.

⑧ Proposed phased plan

Re-sequenced now that the deliverable is a tiered game over the tool. The critical path to a playable game is Phase 2 — the cheap deterministic engine — not the C-lib work. The primer3 spike stays in, but as a thesis-validator and a later Tm-fitness unlock, off the game's critical path.

  1. Phase 0 — this page. Publish the plan, get a green light. ← you are here.
  2. Phase 1 — primer3 spike (off critical path, proves the thesis). A Rust crate that vendors primer3, FFIs oligotm/primer-picking via bindgen+cc, compiled to WASM and called from a static page. Validates "reuse C from Rust → WASM" on the easy lib; later powers Tm-based fitness in directed-evolution and a primer-design mode.
  3. Phase 2 — the engine that powers both games (critical path). ▶ live now. The deterministic subset in Rust → WASM: restriction mapping (30-enzyme table), translation/codon tables + 6-frame ORFs, circular/linear maps, in-silico PCR, digest → gel. Pure compute, no C deps. Built dependency-free (~60 KB wasm), validated against pUC19 (finds its canonical single-cutters; ORF scan recovers the 286-aa β-lactamase). This is the scoring oracle and the workbench core. Try it → · source in splice/engine/
  4. Phase 3 — Assembly-puzzle game. Procedurally generate target constructs + parts libraries; auto-grade validity against the Phase-2 engine; add cost/step optimization + leaderboard histograms. Reuse Fluoddity's score/leaderboard/atlas shell.
  5. Phase 4 — Directed-evolution game (the Fluoddity bridge). Swap Fluoddity's Fourier genome for real DNA; objective fitness functions (codon-optimize, strip sites, GC/Tm); fork + publish constructs as ATProto records; phylogeny + cross-user atlas. Reuses the existing breed→select→fork→gallery loop.
  6. Phase 5 — tier the ramp + real-construct export. Wire the casual puzzle front-end to graduate into the workbench surface; GenBank export you could take to a bench. splice.mino.mobi already deploys "like rite" (Worker + ASSETS, deploy-splice.yml), ready to grow into.
  7. Phase 6 — HMMER (pro-tool only; viability proven). The hardest dependency — its hot loops are hand-written SSE intrinsics that don't exist in WASM — has been spiked and passes. In-browser is viable; server-side is now a choice, not a necessity. See the panel below.

⚑ Spike result: HMMER → WASM is viable (run 2026-05-29)

Before committing to the plan I de-risked the single project-killing unknown — does HMMER's SIMD core survive the trip to WebAssembly? I cloned and built HMMER here, ran its own test suite, and ported its hottest kernel. Result, with code you can re-run (splice/spike/):

CheckResult

So the faithful, in-browser pro-tool tier is real: SIMDe + clang -msimd128 is the porting path, and HMMER's own test suite is the acceptance gate. The only remaining browser cost is profile-DB download size / memory — a data-distribution question, not a feasibility one. Full write-up: spike/VIABILITY.md.

⑨ Decisions

Settled

Game directions: both assembly puzzle and directed evolution on the roadmap (Phases 3 & 4). Both run on the cheap Phase-2 engine.
Audience: tiered — a casual puzzle front-end that graduates into the real workbench (Phase 5).

Still open

How central is HMM search (the /hmmscan surface) to the audience? (Technical viability is now settled — see the spike panel; this is purely a product-priority call: build it, and if so, in-browser or server-side for the profile DB.)
Comfortable that this is a derivative reimplementation of someone else's MIT tool? I'd want a courtesy heads-up + clear attribution to Binomica Labs before anything user-facing ships.
Green light to start building — Phase 2 (the engine) as the critical path, with the Phase 1 primer3 spike alongside to prove the C-reuse thesis?
