Records, Lexicons & Architecture

The Tabard's shared world is backed by a relational database (Postgres) — that is the source of truth for the world pool, replenishment, the rumor mill, the world engine, and the notifications feed. On top of that, a player's own saved game can be mirrored to their own ATProto repo — user-owned, public, $0 to us. This page defines that client/backend split and its contract, then the ATProto mirror: the lexicons, the service account, and how anyone reads the records.

Architecture — the client ↔ backend split

There is a database, by design (a relational store is the right tool for the world engine and the rumor mill). The split is clean, and the client never has to think about the DB:

SideOwns
Backend · Postgres + a service tier The world pool, content replenishment, the rumor mill, the world engine (tick / cascade), and the notifications/retcon feed. The heavy, global, async, and LLM work — all offline of the player hot path.
Client · your browser A deterministic, inference-free engine that dispatches + dedups: it holds the pool plus your personal seen set and decides what crystallizes where. No round-trip per step; works offline; the same engine whether the pool came from Postgres, an API, or an ATProto repo.

The contract — three reads + one feed

WhatShape
the world poolan array of content items (id · type · tags · tiers · content · requires). The backend owns it; the client loads it (already deduped against your seen server-side, or deduped client-side — see below).
your character foldeverything you've done: seen_ids + placements (your frozen feature → content bindings) + facts + inventory + tiers/xp. Delivered as one aggregate read or as an event stream the client folds — the engine is deterministic, so either yields the same in-memory state.
the notifications / retcon feedcom.minomobi.hoop.story.verdict records on the service repo, tagged with your DID — "the Tabard changed under your feet": rumor-mill pushes and entity diffs. The client fetches its own at load + subscribes live (the engine can't write into your repo, so these live on the service repo).

The dedup keystone

The client already does the thing the backend needs: every dispatch computes pool − what you've seen. Each character carries a seen_ids set; selection is pool.filter(c => !seen.has(c.id)) → gate on facts/items → tier-bias → pick → mark seen. First touch crystallizes (freezes a feature → content_id binding); later touches recall it. So "fetch all character events at load, strip the seen from the pool, dedup in memory" is the engine — not a new mechanism.

Retcons — mostly free, by design

Bindings are by id, so the backend's golden rule (edit entities in place — same content id, never swap) makes most retcons free:

Backend actionClient effect
edit an entity in place (enrich / regen, same id)recall returns the new body automatically (placement stores the id; content is looked up live). The notifications feed carries the diff, so the player sees what changed without wandering back.
retire an entity (status ≠ active)recall flags it retired → render as a faded / redacted memory.
force a re-roll / un-see a bindingthe one case needing a push: the feed sends an invalidation (content ids / feature keys to evict from this character's placements + seen); the slot re-crystallizes on next touch.

Where ATProto fits

Postgres is the source of truth; ATProto is an optional mirror of your save — user-owned, public, durable, $0 to us. Drop it and the game is unchanged; keep it and anyone can read your progress (below). The shared world content can likewise be published to the service repo as a public, rebuildable projection. The records below are that mirror layer.

Schemas live in the repo at hoop/lexicons/*.json. The save's stateJson is a serialized MemoryStore.snapshot(), so the save can evolve without re-versioning the lexicon.

Publish spec — the exact record shapes

Enough to start publishing. Rules for all of them: every record carries $type = its NSID; you putRecord(collection, rkey, value) to the repo named below; reads are public, writes are signed by that repo's key. Edit in place — re-putRecord the same rkey; for content, retire via status:"retired", never deleteRecord (a deletion orphans players' crystallized bindings). The three the ENGINE publishes (to morphyx), then the one the CLIENT publishes (to each player's repo) for your consume side.

  "updatedAt": "2026-06-19T12:00:00.000Z"
}

stateJson decode: it's the engine snapshot (single-player, so each [[id,…]] array has one entry). players[0][1] = tiers/xp/seen; facts[0][1] = [[key,value]] incl. character; placements ref content by content_item_id; equipment is slot→invId resolved in inv. Key player identity on the commit DID, not any field inside (the inner "local" is ignored).

Write-only from the client. The game never reads its own rumors; it just appends them. The engine tails this collection across player repos (Jetstream) and may answer by writing a verdict (a retcon/notice, possibly to other players) or new shared content — both channels the client already consumes. No client-side rumor consumer exists by design (cross-player effects flow back through the verdict feed; direct player-vs-player is a later surface).

How an onlooker reads the records

Reads are public and unauthenticated — standard com.atproto.repo XRPC. Three steps:

1. handle → DID → PDS

# resolve a handle to a DID
curl "https://public.api.bsky.app/xrpc/com.atproto.identity.resolveHandle?handle=alice.bsky.social"
# resolve the DID to its PDS (the service endpoint in the DID doc)
curl "https://plc.directory/did:plc:xxxx" | jq '.service[] | select(.id=="#atproto_pds").serviceEndpoint'

2. read one record (a save)

curl "<pds>/xrpc/com.atproto.repo.getRecord?repo=<did>\
&collection=com.minomobi.hoop.story.save&rkey=<world-seed>"

3. list a collection (the shared pool on morphyx)

curl "https://chalciporus.us-west.host.bsky.network/xrpc/com.atproto.repo.listRecords?\
repo=did:plc:yivyyp54vddf7qf2lpsikhe4&collection=com.minomobi.hoop.story.content&limit=100"

An at-uri names any record exactly: at://<did>/<nsid>/<rkey> — e.g. your save is at://<your-did>/com.minomobi.hoop.story.save/<seed>.

Live: read a player's save

Does the round trip in your browser — resolve a handle, find its PDS, and fetch any com.minomobi.hoop.story.save records there. (Empty if that person hasn't signed into the game.)

Why this shape? See the data-flow and the story spine. The rule that matters: truth is the records; everything else is a cache.
