Records & Lexicons

The Tabard has no database. Every durable thing — the shared world content, the cross-player pulse, and your own saved game — is an ATProto record in a repo. Caches (the in-memory engine, localStorage) are disposable projections; the truth is the records, and the records are public. This page defines the lexicons in play, names the service account, and shows anyone how to read them.

The service account

morphyx — the Tabard's service repo (the same account borges uses).
handle morphyxmino.bsky.social

morphyx hosts the shared, read-mostly collections: the world's content pool and the Director's pulse. It never holds your save — that lives in your repo, paid for by you, owned by you.

The lexicons

NSIDLives inrkeyWhat it is
com.minomobi.hoop.story.savethe player's repoworld key (ship seed) Your durable game state for one world — facts, placements, inventory, equipment, NPC standing, xp & tiers — as a serialized snapshot (stateJson). Written batched (debounced, on unload), never per footstep.
com.minomobi.hoop.story.contentmorphyxcontent id One world entity (creature / item / lore_fragment / npc / plot_beat / rumor), stored verbatim. The shared pool the engine crystallizes from; a player's own generated side-quests can be written to their repo under the same NSID.
com.minomobi.hoop.story.pulsemorphyxself The Director's cross-player rollup — how many travellers have met what / been where — folded from player saves off the firehose. A disposable projection, rebuildable by replaying Jetstream.
com.minomobi.hoop.placethe player's repotile / chamber addr A place on the map that anchors a thread (the main-site forum layer).
com.minomobi.hoop.messagethe player's repotid A message in a place's thread.

Schemas live in the repo at hoop/lexicons/*.json. The save's stateJson is a serialized MemoryStore.snapshot(), so the save can evolve without re-versioning the lexicon.

How an onlooker reads the records

Reads are public and unauthenticated — standard com.atproto.repo XRPC. Three steps:

1. handle → DID → PDS

# resolve a handle to a DID
curl "https://public.api.bsky.app/xrpc/com.atproto.identity.resolveHandle?handle=alice.bsky.social"
# resolve the DID to its PDS (the service endpoint in the DID doc)
curl "https://plc.directory/did:plc:xxxx" | jq '.service[] | select(.id=="#atproto_pds").serviceEndpoint'

2. read one record (a save)

curl "<pds>/xrpc/com.atproto.repo.getRecord?repo=<did>\
&collection=com.minomobi.hoop.story.save&rkey=<world-seed>"

3. list a collection (the shared pool on morphyx)

curl "https://chalciporus.us-west.host.bsky.network/xrpc/com.atproto.repo.listRecords?\
repo=did:plc:yivyyp54vddf7qf2lpsikhe4&collection=com.minomobi.hoop.story.content&limit=100"

An at-uri names any record exactly: at://<did>/<nsid>/<rkey> — e.g. your save is at://<your-did>/com.minomobi.hoop.story.save/<seed>.

Live: read a player's save

Does the round trip in your browser — resolve a handle, find its PDS, and fetch any com.minomobi.hoop.story.save records there. (Empty if that person hasn't signed into the game.)

Why this shape? See the data-flow and the story spine. The rule that matters: truth is the records; everything else is a cache.
