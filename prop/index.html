A rule rewrites the graph. It says: wherever this little pattern of links appears, delete it and put this other pattern there instead — minting fresh nodes for any variable on the right that wasn't on the left. Fire the rule again and again and the graph builds itself.

notation

A link is a tuple of nodes in braces; a pattern is a set of links; lowercase letters are variables. The left of → is what to match, the right is what to leave behind.

{{x,y}} → {{x,z},{z,y}} reads: find any link x–y; replace it with two links through a brand-new node z. That single rule subdivides — every link it touches becomes a longer chain.

Variables only on the right (z, w) are fresh nodes — this is how the graph grows. Variables on both sides are re-used, stitching the new structure onto the old. A self-link {x,x} is allowed (it renders as a small loop) and is a handy seed.

You described a force-directed graph that periodically sprouts a new node, which then “interacts down the chain” Game-of-Life-style, sometimes spawning more nodes and links. That’s a real and well-studied idea — but it travels under several names, and the word propagator points at a slightly different one. Here’s the lay of the land.

what you most likely saw

The viral “growing graph” videos are almost always hypergraph rewriting, popularised by the Wolfram Physics Project (2020). The whole state of the world is a graph; a fixed local rewrite rule says “wherever this small sub-pattern occurs, replace it with that one.” Applying it over and over makes the graph grow and fold, and with a force-directed layout it looks like a living, branching organism. That is exactly the engine in the play tab.

It is genuinely Game-of-Life-like: Conway’s Life is a rewrite rule on a fixed grid; this is a rewrite rule on a graph whose shape itself changes as the rule fires. Same spirit — local rule, global emergent behaviour — but the substrate grows.

but “propagators” is its own field

The literal term propagator networks comes from Gerald Sussman and Alexey Radul’s “The Art of the Propagator” (2009) and Radul’s MIT thesis “Propagation Networks”. It is a computation model, not a growth model:

Cells accumulate information; propagators are stateless machines wired between cells that wake up whenever a neighbouring cell learns something new, compute, and push refined information back out — in any direction.

So a propagator network is a fixed graph where information flows and merges (it underlies constraint solving, truth-maintenance, and spreadsheet-like reactive systems), whereas the growing-graph videos are about structure being rewritten. They’re cousins — both are “local machines firing on local changes, chained across a network” — which is probably why the same video gets called by either name. This playground builds the structural-growth kind, since that’s what you described visually; the table below places both.

best practices (yes, there are some)

Building one of these well comes down to a handful of decisions the literature has converged on:

  • The updating scheme is a real choice. When several matches overlap, which fire? Wolfram’s standard is a maximal non-overlapping set per “generation”; the alternative is one random match at a time. The play tab exposes both (sequential vs generations) because they produce visibly different geometry from the same rule.
  • Confluence & causal invariance. If the final result doesn’t depend on the order matches fire, the system is confluent (Wolfram calls the graph version causal invariance). Interaction nets are confluent by construction — a celebrated property. Most interesting rules aren’t, which is exactly why order matters above.
  • Subgraph matching is the bottleneck. Finding where a rule applies is subgraph isomorphism — NP-hard in general. Real engines index by node degree/labels and match incrementally; a playground just caps the graph size (the node cap slider) and randomises match order for variety.
  • Layout is a separate problem from rewriting. The standard answer is a force-directed simulation — links are springs, all nodes repel (Fruchterman–Reingold / Barnes–Hut for big graphs). New nodes are dropped near their parents so growth looks organic rather than teleporting. Re-layout continuously; don’t freeze.
  • Determinism where you can get it. Seed the RNG so a given rule + seed replays identically — essential for sharing and debugging. (Here, “random match” is the one deliberate exception.)
  • Termination. Most growth rules never halt, so you bound them: a node cap, a generation count, or an energy threshold. Without a bound the tab would happily melt your laptop.
